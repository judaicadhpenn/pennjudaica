# Harvesting `data.js` from Penn's source systems

The front end never talks to Penn's systems directly — it reads one file,
`data.js`, which exposes `window.PJP = { sources, items, featured }`. In the
beta that file is hand-built. In production it should be **generated by a
server-side harvester** that pulls from Penn's real systems and writes the same
shape. This document describes that pipeline.

## Why harvest server-side (and not fetch in the browser)

A static page can freely load **images** cross-origin (that's why the OPenn
images work here), but it generally **cannot fetch metadata** from another
origin: OPenn, the Franklin catalog, Colenda, and the finding-aids server don't
send permissive `Access-Control-Allow-Origin` headers, so an in-browser
`fetch()` is blocked by CORS. The portable, durable pattern is:

```
  Penn source systems                Harvester (server-side, scheduled)        Front end
  ───────────────────                ──────────────────────────────────        ─────────
  Franklin catalog (MARC) ─┐
  Colenda (OAI-PMH/MODS) ──┼──▶ normalize ▶ map to item schema ▶ enrich ▶ ──▶ data.js  (or data.json
  OPenn (TEI + IIIF) ──────┤        with IIIF manifests (pages, deep zoom)        + a small search API)
  Finding aids (EAD) ──────┘
```

Run the harvester on a cron (nightly/weekly) or trigger it from
[IIIF Change Discovery](https://iiif.io/api/discovery/) activity streams, then
publish a fresh `data.js`. At real collection sizes, write the records into a
search index (Solr/Elasticsearch — Franklin already runs Solr via Blacklight)
and have the front end query a thin read-only API instead of a flat file. The
UI code does not change; only the source of `window.PJP.items` does.

## The two protocols

### 1. OAI-PMH — for metadata

OAI-PMH is the harvesting standard most Penn discovery systems can speak
(Colenda/Hyrax and most ArchivesSpace/EAD pipelines expose it; the catalog can
provide MARCXML exports). You repeatedly call `ListRecords` and page through
with a `resumptionToken`.

Useful verbs: `Identify`, `ListMetadataFormats`, `ListSets`,
`ListRecords` (bulk), `GetRecord` (one). Request a rich metadata prefix when
available (`mods`, `marc21`) rather than only `oai_dc`, so you keep language,
place, genre, and date granularity.

### 2. IIIF Presentation API — for images, pages, and deep zoom

For anything digitized, resolve its **IIIF Presentation manifest**. A manifest's
`items` (Presentation 3) / `sequences[].canvases` (Presentation 2) are the
**leaves** — map each canvas to one entry in the item's `pages[]`. Each canvas
points to an **IIIF Image API** service; that service URL is what you hand to a
deep-zoom viewer (OpenSeadragon, or Samvera's Clover) for true tile-based zoom —
better than the single web JPEG this beta uses.

```
manifest.items[n]                      -> pages[n]
  canvas.thumbnail[0].id               -> pages[n].thumb
  canvas ▸ annotation ▸ body.service   -> IIIF Image API base
     <image-service>/full/max/0/default.jpg   -> pages[n].img   (or tile source)
  canvas.label                         -> pages[n].label
```

> The beta uses OpenSeadragon in single-image mode. To switch to real IIIF deep
> zoom, change the tile source in `app.js → loadImage()` from
> `{ type:"image", url: pg.img }` to the IIIF service, e.g.
> `pg.service + "/info.json"`. OpenSeadragon reads `info.json` natively.

## Target schema (what the harvester must emit)

Each `items[]` entry:

| field | source mapping (example) |
|-------|--------------------------|
| `id` | stable local id / hashed identifier |
| `title` | MODS `titleInfo` / MARC 245 / DC `title` |
| `creator` | MODS `name` / MARC 1xx,7xx / DC `creator` |
| `year`, `dateText` | MODS `originInfo/dateCreated` (parse to int + keep display string) |
| `place`, `region`, `lat`, `lng` | MODS `originInfo/place`; geocode via GeoNames or Getty TGN |
| `lang` | MODS `language` / MARC 008,041 |
| `type` | MODS `typeOfResource`/`genre` / MARC LDR |
| `collection`, `source` | OAI set / repository id |
| `callno` | MODS `identifier` / MARC 099 |
| `srcUrl` | landing-page URL in the source system |
| `img` | representative thumbnail (a IIIF thumbnail) |
| `pages[]` | one per IIIF canvas: `{ img, thumb, label }` |
| `desc` | MODS `abstract`/`note` / DC `description` |

## Sample harvester (Python)

Dependencies: `pip install sickle requests`. This sketch harvests one OAI set,
resolves each record's IIIF manifest, and writes `data.js`.

```python
import json, re, requests
from sickle import Sickle

OAI   = "https://<penn-oai-endpoint>/oai"      # e.g. Colenda OAI-PMH
PREFIX = "mods"                                 # prefer mods/marc21 over oai_dc
SETS  = {"zucker": "ketubah", "ljs": "schoenberg", "mikveh": "mikveh_israel"}

def manifest_pages(manifest_url):
    """Return [{img, thumb, label}] from a IIIF Presentation manifest."""
    m = requests.get(manifest_url, timeout=30).json()
    pages = []
    canvases = m.get("items") or (m.get("sequences", [{}])[0].get("canvases", []))
    for c in canvases:
        # Presentation 3
        label = (c.get("label", {}) or {}).get("none", [""])[0] if isinstance(c.get("label"), dict) else c.get("label", "")
        try:
            body = c["items"][0]["items"][0]["body"]
            svc  = (body.get("service") or [{}])[0]
            base = svc.get("id") or svc.get("@id") or body["id"]
            img  = base + "/full/1140,/0/default.jpg" if "/full/" not in base else base
            thumb = base + "/full/120,/0/default.jpg"
        except Exception:
            img = c.get("thumbnail", [{}])[0].get("id", ""); thumb = img
        pages.append({"img": img, "thumb": thumb, "label": label})
    return pages

def to_item(rec, source_key):
    md = rec.metadata  # dict of MODS fields from sickle
    return {
        "id": rec.header.identifier,
        "title": (md.get("title") or [""])[0],
        "creator": (md.get("name") or ["Unknown"])[0],
        "year": int((re.findall(r"\d{4}", (md.get("dateCreated") or [""])[0]) or ["0"])[0]),
        "dateText": (md.get("dateCreated") or [""])[0],
        "place": (md.get("place") or [""])[0],
        "region": "",            # derive/geocode in an enrich step
        "lat": None, "lng": None,
        "lang": (md.get("language") or [""])[0],
        "type": (md.get("genre") or md.get("typeOfResource") or [""])[0],
        "collection": (md.get("relatedItem") or [""])[0],
        "source": source_key,
        "callno": (md.get("identifier") or [""])[0],
        "srcUrl": (md.get("location") or [""])[0],
        "desc": (md.get("abstract") or md.get("note") or [""])[0],
        "img": "", "pages": [],
    }

def harvest():
    s = Sickle(OAI)
    items, idn = [], 0
    for key, oai_set in SETS.items():
        for rec in s.ListRecords(metadataPrefix=PREFIX, set=oai_set, ignore_deleted=True):
            it = to_item(rec, key); idn += 1; it["id"] = idn
            manifest = it.get("srcUrl", "")  # or a dedicated manifest field
            if manifest.endswith("manifest") or "manifest" in manifest:
                it["pages"] = manifest_pages(manifest)
                if it["pages"]:
                    it["img"] = it["pages"][0]["thumb"] or it["pages"][0]["img"]
            items.append(it)
    sources = {
        "zucker": {"label": "Zucker Ketubah Collection", "system": "OPenn", "color": "#a8492b"},
        "ljs":    {"label": "Schoenberg Manuscripts",     "system": "OPenn", "color": "#6e7637"},
        "mikveh": {"label": "Mikveh Israel Records",      "system": "OPenn", "color": "#6a4a7c"},
    }
    payload = {"sources": sources, "items": items, "featured": [i["id"] for i in items[:4]]}
    with open("data.js", "w") as f:
        f.write("/* generated %s */\nwindow.PJP = %s;\n" % (__import__("datetime").date.today(), json.dumps(payload, ensure_ascii=False, indent=2)))

if __name__ == "__main__":
    harvest()
```

This is intentionally a skeleton: in practice add a geocoding/enrichment pass
(region + lat/lng from place authorities), de-duplication across systems (the
same item may appear in both the catalog and Colenda — merge on identifier), and
validation before publishing. EAD finding aids are harvested the same way but
mapped at the component level; catalog-only records simply arrive with empty
`pages[]` and the UI shows them as not-yet-digitized.
